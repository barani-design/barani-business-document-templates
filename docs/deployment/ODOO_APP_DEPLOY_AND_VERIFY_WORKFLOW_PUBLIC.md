# Odoo App Deploy and Verification Workflow

This document describes a public, version-neutral workflow for safely deploying and verifying Odoo add-ons. It is designed for teams using a Git repository, a staging/production hosting environment, and post-deploy database/UI verification.

This is a sanitized public version. Replace placeholders such as `<module>`, `<branch>`, `<repo>`, and `<hosting environment>` with your own project details.

## Principles

1. **Separate development and verification.** The person or chat preparing the code should not be the only reviewer. Verify each delivery independently before deployment.
2. **One module, one branch.** Avoid cross-wiring unrelated add-ons or deploying one module from another module's branch.
3. **Use gated deploys.** Do not push until the package, checksum, staged diff, and target branch are verified.
4. **Verify live behavior after a green build.** A green build only proves installation or upgrade completed. It does not prove the feature produced the expected data or UI behavior.
5. **Prefer fix-forward over revert.** If a deployment fails before going live, fix the code and redeploy. If data was changed, evaluate with a backup/restore plan before any rollback.
6. **Never publish private data.** Keep customer exports, database dumps, live XML exports, credentials, and real operational logs out of public repositories.

## Roles

### Developer

Prepares the module/package, migrations, tests, and handoff notes.

### Verifier / deployer

Independently reviews the delivered code/package, checks the staged diff, confirms the deployment target, and verifies the result in the live/staging system.

## Pre-deploy audit checklist

Before touching the deployment branch, review the delivery in a clean environment:

```bash
# Example only: adapt paths to your module/package
python3 -m py_compile $(find <module> -name '*.py')
find <module> \( -name '*.pyc' -o -name '__pycache__' -o -name '.DS_Store' \) -print
```

Check:

- module structure and manifest;
- migrations are idempotent and non-destructive;
- XML/QWeb files parse where possible;
- no generated bytecode or local cache files;
- no credentials, private exports, or customer data;
- version number matches the intended deployment.

## Checksum gate

If deploying from a zip/package, record a checksum during audit and verify the exact same file before deployment.

Linux:

```bash
sha256sum ~/Downloads/<delivery>.zip
```

macOS:

```bash
shasum -a 256 ~/Downloads/<delivery>.zip
```

The audited checksum must equal the deployed checksum.

## Branch and working-tree gate

From the local checkout:

```bash
cd <repo>
git switch <branch>
git status
git log --oneline -3
```

Expected:

- correct branch;
- clean working tree before swapping files;
- local branch up to date with remote;
- current version matches the deployed baseline.

## Swap package and review staged diff

Example for one module folder:

```bash
cd <repo>
rm -rf <module>
unzip -q ~/Downloads/<delivery>.zip
grep "'version'" <module>/__manifest__.py
find <module> \( -name '*.pyc' -o -name '__pycache__' -o -name '.DS_Store' \) -print
git add -A
git diff --cached --stat
git diff --cached -- <module>
```

Only expected files under the intended module should appear. If unrelated folders/files appear, stop and reconcile before committing.

## Commit

Use a clear commit message that includes scope and package/version reference.

```bash
git commit -m "Deploy <module> <version>" -m "Scope: <short summary>. Checksum: <sha256>."
```

## Push

Use your team's approved push method. If using GitHub Desktop, confirm:

- current repository;
- current branch;
- no unrelated local changes;
- push target is correct.

## Hosting build verification

After push:

1. Watch the new build for the correct commit.
2. Do not rebuild unless the hosting platform documentation and your data-retention plan explicitly require it.
3. Read the relevant build logs if the build fails.
4. Confirm the new build became live before running verification commands.

## Server/database verification

Open a fresh server shell after the build. For SQL checks, disable paging:

```bash
export PAGER=cat
```

Confirm module install state:

```bash
psql -c "SELECT name, state, latest_version FROM ir_module_module WHERE name='<module>';"
```

Then verify the feature-specific data. For example:

```bash
psql -c "SELECT COUNT(*) FROM <expected_table>;"
psql -c "SELECT id, create_date, write_date FROM <expected_table> ORDER BY id DESC LIMIT 5;"
```

Be careful with stale data. A green build does not recompute old records. To verify new behavior, create a fresh test record or rerun the supported computation/action.

## UI verification

After the database checks, verify the feature in the Odoo UI:

- menus are visible to the correct group;
- forms open without traceback;
- reports/actions render;
- generated data matches the expected business rule;
- no unrelated modules/regressions are visible.

## Public-repository hygiene

Do not commit:

- production database dumps;
- customer PDFs/invoices/exports;
- live XML exports containing company/customer data;
- private branch names or deployment URLs;
- credentials, tokens, API keys, server-shell logs;
- screenshots containing customer or bank information.

## Minimal deployment runbook

1. Audit package in clean sandbox.
2. Record checksum.
3. Confirm branch and clean working tree.
4. Verify checksum locally.
5. Swap the module package.
6. Stage and inspect diff.
7. Commit.
8. Push using approved method.
9. Watch build.
10. Verify module state and feature output in database/UI.
11. Document result.
